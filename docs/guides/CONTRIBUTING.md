## 1. Измените статус задачи в [Trello](https://trello.com/b/dVBBwC4t/%D0%B7%D0%B0%D0%BC%D0%BE%D1%80%D1%8B%D1%88)

Переместите карточку задачи в колонку `In Progress`.

## 2. Синхронизируйтесь с GitHub и создайте новую ветку

1. Перед началом работы в терминале убедитесь, что вы находитесь в директории проекта. Если вы находитесь в родительской
   папке, перейдите в директорию проекта:
   
    ```bash
    cd zamorysh
    ```
    
2. Перейдите на ветку main и синхронизируйтесь с GitHub:

    ```bash
    git checkout main
    git pull
    ```
    
3. Создайте новую ветку для задачи, перейдите на нее:

    ```bash
    git checkout -b feature-name
    ```

## 3. Работайте в этой ветке

1. Добавьте файлы в коммит (можно все, можно только те, изменения в которых вы хотите сохранить):

    ```bash
    git add .
    ```
    
    ```bash
    git add path/to/file.py
    ```
    
2. Перед тем как сделать коммит, проверьте, что именно вы коммитите:

    ```bash
    git status
    ```

3. Сделайте коммит. Описание коммита должно быть коротким предложением на английском языке и начинаться с глагола в
   императиве.
   
    ```bash
    git commit -m "Add commit name here"
    ```
    
4. Повторяйте, пока не закончите работу над задачей или пока у вас не возникнут вопросы.

<a name="github-sync"></a>

5. Периодически синхронизируйтесь с GitHub. Для этого перейдите на ветку main и обновите ее:

    ```bash
    git checkout main
    git pull
    ```
    
6. Перейдите на рабочую ветку и влейте в нее обновленную ветку main:

    ```bash
    git checkout feature-name  
    git merge main
    ```

7. При наличии конфликтов разрешите их.

## 4. Говорите о проблемах

### Если есть вопросы

Если ваш код не запускается или падает во время выполнения, сначала попробуйте разобраться самостоятельно. Посмотрите на
сообщение об ошибке, поищите ответ на StackOverflow, спросите нейросеть.

Если код работает, но не так, как вы бы хотели, включите логирование и посмотрите логи, при необходимости добавьте свои.

Если за полчаса у вас не получилось разобраться самостоятельно, пожалуйста, задайте вопрос.
Вопрос вроде «Почему мой код не работает?» — тоже отличный вопрос.
На примере вашего кода я смогу объяснить вам именно то, что нужно для вашей задачи.

Если вы не понимаете, с чего начать и что делать, тоже задайте вопрос.

Задать вопрос можно двумя способами: можно написать мне в личку, можно написать в чат. Перед тем как задать вопрос,
подумайте, может ли ответ быть полезен другим участникам, и действуйте в соответствии с этим. Я постараюсь ответить на
вопрос в течение 24 часов.

Чтобы я смог увидеть ваш код, загрузите изменения на GitHub, не создавая pull request (см. [6.1 — 6.2](#github-push)).

<a name="no-time"></a>
### Если вы не успеваете

В этом нет ничего страшного. Возможно, у задачи слишком расплывчатое ТЗ или задача слишком объемная.

Разделите вашу задачу на подзадачи. Упростите задачу, оставив только самое важное. Остальное реализуете, если останется время.
Сделать что-то гораздо лучше, чем не сделать ничего.

Создайте карточку для упрощенной версии задачи: скопируйте исходную карточку, измените ее название и при необходимости описание.
Далее работайте с новой версией задачи по тому же алгоритму, что и с исходной.

Если вы понимаете, что не успеете реализовать и упрощенную версию задачи, всё равно упростите задачу.

### Если ваша задача состоит из нескольких этапов

Например, сначала Вам нужно придумать, как что-то реализовать, а потом уже написать код. В таком случае сообщайте мне
о том, что вы сделали, после каждого этапа, чтобы в случае чего не тратить время зря.

## 5. Протестируйте новые функции

Перед тем как создавать pull request, проверьте, что всё работает так, как вы задумывали.

Если вы создавали контент, проверьте, правильно ли он работает в игре. Включите логирование.

Если вы создавали API или какую-то систему, создайте какой-нибудь тестовый файл и повызывайте оттуда ваши функции (только не забудьте потом его удалить и не добавляйте его в Git).

## 6. Загрузите изменения на GitHub и создайте pull request

<a name="github-push"></a>
1. Синхронизируйтесь с GitHub (см. [3.5 — 3.7](#github-sync)).

2. Укажите, в какую ветку загружать изменения (это должна быть ветка в удаленном репозитории, то есть origin, с тем же названием, что и ваша локальная), и загрузите их:

    ```bash
    git push -u origin feature-name
    ```
    
   Если вы уже загружали изменения из текущей ветки, можно не указывать:
   
    ```bash
    git push
    ```

3. Если вы загружаете изменения из текущей ветки впервые, в терминале появится ссылка для создания pull request.
   Перейдите по ней, напишите название и описание при необходимости, создайте pull request.
   В противном случае создать pull request можно через браузер, открыв репозиторий.

## 7. Измените статус задачи в [Trello](https://trello.com/b/dVBBwC4t/%D0%B7%D0%B0%D0%BC%D0%BE%D1%80%D1%8B%D1%88)

Переместите карточку задачи в колонку `In Review`.

## 8. При необходимости внесите изменения

Если у меня будут какие-то замечания, я оставлю на странице pull request комментарии и сообщу вам об этом. Чтобы внести изменения, сделайте коммит, а потом загрузите изменения
на GitHub.

Когда вы внесете все необходимые правки, я сделаю merge, перемещу карточку в колонку `Done` и отмечу задачу как выполненную.

## 9. После merge удалите ветку из локального репозитория

Перейдите на main и удалите старую ветку:

```bash
git checkout main
git branch -d feature-name
```

Из репозитория на GitHub ветка будет удалена автоматически.

## 10. Приходите на созвон

### I. Если вы выполнили задачу или ее часть

Расскажите другим о своей работе, покажите и разберите свой код.

### II. Если вы не успели выполнить задачу и при этом вам что-то непонятно

Заранее сформулируйте и задайте на созвоне любой вопрос. Вопрос может касаться как вашего собственного кода, так и чужого, может как напрямую относиться к вашей задаче,
так и не относиться к ней.

### III. Если вы не успели выполнить задачу, но при этом вам всё понятно

Убедитесь, что вам действительно всё понятно. Откройте код движка и попробуйте найти какое-нибудь непонятное место.
Если вы зададите вопрос, я смогу на него ответить, поэтому это гораздо лучше, чем ничего.

Расскажите, как выглядит упрощенная версия вашей задачи (см. [4. Если вы не успеваете](#no-time)), и выполните эту упрощенную версию к следующему созвону.
